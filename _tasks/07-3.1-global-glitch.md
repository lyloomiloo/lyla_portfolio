# Task: Upgrade GlitchText for multiple instances with staggered timing
Plan: `/_plans/07-design-refinements.md` → Step 3.1
Spec: `/_specs/07-design-refinements.md`

## Context
Currently glitch only runs on the hero title. It should run continuously on multiple elements across the page, each on its own random timer.

## What to build
Modify GlitchText to support many simultaneous instances without synchronization.

## Technical details

### New props
- `intensity`: "subtle" (1-2 chars) | "bold" (2-4 chars) — default "bold"
- `interval`: number in ms — base interval, will be randomized ±30%. Default 3500.
- `tag`: HTML tag to render. Default "span".
- `class`: pass-through class.

### Staggering logic
Each instance should:
1. Generate a random initial delay (0 to interval ms) on mount
2. Start its timer after the initial delay
3. Each tick, randomize the next interval by ±30% of the base

This means if you have 6 GlitchText instances on the page, they'll all glitch at different times, creating a constant ambient effect without being synchronized.

### Character set
Keep: `* # @ / \ ] [ 0 1 { } < > _ | ~`

### The glitch effect
Same as before — pick random non-space characters, replace with symbols, restore after 80-120ms. The `intensity` prop controls how many characters are affected.

### CSS glitch on load
The initial CSS clip-path glitch should ONLY play on the hero title, not on every instance. Add a `playEntrance` boolean prop (default false). Only the hero sets this to true.

## Files to create/modify
- `src/components/GlitchText.astro` — rewrite with new props and staggering

## Acceptance criteria
- Multiple GlitchText instances on the page glitch independently
- Timing is staggered — no two elements glitch at the same moment
- "subtle" intensity affects 1-2 chars, "bold" affects 2-4
- CSS entrance animation only plays on hero
- No performance issues with 6+ instances
